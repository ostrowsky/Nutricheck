# Спецификация: NutriCheck Python Analyzer (Task 2)

## Цель продукта
Консольный (CLI) инструмент на Python, упакованный в Docker, для глубокого анализа дефицитов рациона на основе CSV-дневника питания и автоматического обогащения нутриентов из Open Food Facts API.

## Входные данные (Input Contract)
1. **CSV-дневник питания**: Таблица, содержащая как минимум две колонки:
   - Название продукта (кириллица/латиница).
   - Вес порции в граммах.
   - Поддерживаемые разделители: `;` или `,`.
2. **Аргументы командной строки (CLI arguments)**:
   - `-i` / `--input`: Путь к входному CSV (default: `data/sample_food_log.csv`).
   - `-g` / `--gender`: Пол пользователя (`male` или `female`).
   - `-a` / `--age`: Возраст пользователя в годах (integer).
   - `-act` / `--activity`: Активность (`sedentary`, `moderate`, `active`).
   - `--csv-output`: Путь сохранения CSV-отчета (default: `output/report.csv`).
   - `--json-output`: Путь сохранения JSON-отчета (default: `output/report.json`).

## Бизнес-логика (Invariants)
- **API-кэширование**: Повторяющиеся в CSV-файле названия продуктов не должны инициировать новые запросы в API; данные берутся из локального in-memory кэша.
- **Отказоустойчивость**: При временной недоступности API Open Food Facts (ошибки 500, 503, 504) или сбоях сети скрипт делает до 3-х повторных попыток (retries) с экспоненциальной задержкой.
- **Расчёт дефицитов**: Вычисляются критические дефициты (<50% суточной нормы) и умеренные дефициты (50-80% суточной нормы).
- **Экспорт данных**: Генерируемый `report.json` должен строго соответствовать формату импорта Задачи 1.

## Граничные случаи (Edge Cases)
- **Терминалы без поддержки Unicode на Windows**: Если кодировка консоли не поддерживает вывод эмодзи (UnicodeEncodeError), поток переключается в UTF-8.
- **Продукт не найден в API**: Если продукт не найден, скрипт ставит статус `⚠️ Не найден в API` в консоли, не прерывая общую обработку.
